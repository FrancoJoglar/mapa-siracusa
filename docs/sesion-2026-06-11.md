# Sesión 2026-06-11 — Resumen de problemas solucionados

## 1. Error HTTP 400 — "Geometry has Z dimension but column does not"

**Síntoma:** Al guardar un polígono editado con 3+ partes, PostGIS rechazaba la geometría con error `22023`.

**Causa raíz:** Las coordenadas extraídas por `extractCoordsFromLayer` contenían un tercer valor Z (altitud) que PostGIS no aceptaba en columnas definidas como 2D (`Geometry, 4326`).

**Solución:** Se creó la función SQL `update_sector_geom` que envuelve la geometría con `ST_Force2D()` antes de guardar:

```sql
CREATE OR REPLACE FUNCTION update_sector_geom(p_id UUID, p_geojson JSONB)
RETURNS void LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  UPDATE sectores SET geometria = ST_Force2D(ST_GeomFromGeoJSON(p_geojson::text))
  WHERE id = p_id;
END;
$$;
```

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — `doSave()` usa RPC `update_sector_geom` para sectores
- SQL ejecutado directo en Supabase via Management API

## 2. Error al eliminar polígono — se borraban TODOS en vez de uno

**Síntoma:** Al usar la herramienta de eliminar de Geoman (o el botón custom), clickear un polígono eliminaba TODOS los polígonos del sector.

**Causa raíz:** `L.geoJSON(initialGeoJSON)` crea un `FeatureGroup` que agrupa todos los polígonos hijos. Al eliminar uno con Geoman, el grupo eliminaba todos sus hijos.

**Solución:** Se reemplazó `L.geoJSON` por creación manual de `L.polygon()` individuales, completamente independientes entre sí. No hay grupo padre que pueda propagar la eliminación.

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — init cambió de `L.geoJSON().eachLayer()` a loop manual que crea `L.polygon(latlngs)` independientes

## 3. Polígono invertido (el área se mostraba al revés)

**Síntoma:** Al dibujar un polígono, el área interior se mostraba como vacío y el exterior como polígono.

**Causa raíz:** Los anillos estaban en sentido horario (CW) en vez de antihorario (CCW). PostGIS/Leaflet interpreta CW como "hueco" (exterior) e invierte el área.

**Solución 1 (preventiva):** Se agregó en `extractCoordsFromLayer` una verificación de orientación usando la fórmula del área signada (shoelace). Si el área es negativa (CW), se revierte el anillo:

```ts
rings = rings.map(ring => {
  let area = 0;
  for (let i = 0; i < ring.length; i++) {
    const j = (i + 1) % ring.length;
    area += ring[i][0] * ring[j][1] - ring[j][0] * ring[i][1];
  }
  return area > 0 ? ring : ring.slice().reverse();
});
```

**Solución 2 (correctiva):** Se ejecutó migración para corregir los sectores E1S3 y E1S4 que tenían anillos CW.

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — `extractCoordsFromLayer`
- SQL ejecutado directo en Supabase

## 4. Polígono fantasma en medio del océano (coordenadas [0,0])

**Síntoma:** Aparecía un polígono con coordenadas `[0, 0]` (Golfo de Guinea, cerca de África) al guardar, haciendo que el mapa se alejara al abrir el editor.

**Causa raíz:** Geoman generaba polígonos temporales/fantasma durante la edición que quedaban trackeados en `polyLayersRef` y se guardaban en la DB.

**Solución:** Se agregó un filtro en el save que rechaza polígonos con coordenadas cercanas a `[0, 0]`:

```ts
const hasBad = allCoords.some(c => Math.abs(c[0]) < 0.001 && Math.abs(c[1]) < 0.001);
const validPoints = allCoords.filter(c => Math.abs(c[0]) > 0.001 || Math.abs(c[1]) > 0.001).length;
if (hasBad || validPoints < 3) { /* skip */ }
```

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — `handleSave`

## 5. Save exitoso pero sin persistencia de cambios

**Síntoma:** El editor mostraba "Polígono guardado correctamente" pero al reabrir los cambios no estaban.

**Causa raíz 1:** `supabase.from(table).update().match()` no devolvía error para status 400 — el catch no detectaba el fallo.

**Causa raíz 2:** `allFeatures` estaba vacío (por filtro `interactive=false`) y el código caía al fallback `doSave(initialGeoJSON)`, guardando la geometría original.

**Solución:** Se reemplazó `supabase.update()` por `fetch()` directo con validación explícita de `resp.ok`. Se reemplazó el filtro `interactive === false` por `polyLayersRef` (trackeo explícito de capas).

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — `doSave()`, `handleSave()`
- Eliminación de `interactive === false` filter
- Reemplazo de `supabase.update()` por `fetch()` directo

## 6. Solo se detectaban 2 de 4 polígonos al guardar

**Síntoma:** El log mostraba "features: 2" cuando el sector tenía 4 polígonos.

**Causa raíz:** `map.eachLayer` iteraba 173 capas, de las cuales 123 tenían `interactive=false` (context layers + posiblemente capas de Geoman). El filtro `interactive === false` eliminaba las capas correctas.

**Solución:** Se reemplazó `map.eachLayer` por `polyLayersRef` — un `Set` que trackea explícitamente las capas que pertenecen al editor:
- Polígonos iniciales cargados desde la DB
- Polígonos dibujados por el usuario (vía `pm:create`)
- Polígonos eliminados se remueven del Set (vía `pm:remove`)

**Archivos modificados:**
- `src/components/map/GeomanEditor.tsx` — nuevo `polyLayersRef`, reemplazo de `eachLayer`
